# Acetone — Roadmap and Implementation Plan

*v0.1, July 2026. Phases are sequential with hard exit criteria; sizes are relative (S/M/L/XL) rather than calendar estimates, on the assumption of agentic implementation via Claude Code with human review at the gates.*

## Guiding principles

Correctness properties are enforced by tests before features accrete: history independence, encoding determinism, merge determinism and TCK conformance are all mechanically checkable, and each phase's exit criteria lean on them. The riskiest assumptions are retired first (Phase 0 exists solely to de-risk the git-as-chunk-store bet before anything is built on it). Every phase ends with something usable from the CLI, however small, because a workbench product should feel like a product early.

## Phase 0 — Feasibility spike (size S)

Build a throwaway prolly map over the git object database: content-defined chunking (~4 KiB target), put/get/scan, root-manifest-in-a-git-commit, using gitoxide with git2 as fallback. Evaluate the existing `prollytree` crate hands-on as part of this — adopt, fork, or treat as reference. Benchmark against the representative asset-registry envelope: 100k, 1M and 5M keys; bulk load, point read, range scan, single-key update (measuring write amplification), diff between adjacent versions, repo size after 100 simulated import commits, and repack behaviour (loose objects vs pack-first writing).

Exit criteria: history independence demonstrated under randomised operation orders (property test); update latency and repo growth acceptable at 1M keys; a written decision on gitoxide vs git2 for the write path and on adopt-vs-build for the tree; go/no-go on Decision 1 Option A. If no-go, pivot to the hybrid Option C design before proceeding — nothing above this layer changes either way, which is the point of doing this first.

## Phase 1 — Storage and model core (size L)

Implement `acetone-store`, `acetone-prolly` and `acetone-model` per spec §3: the ChunkStore trait and git backend; production prolly trees with diff and three-way merge; order-preserving key encoding and canonical CBOR values with exhaustive round-trip and ordering property tests; schema map, node/edge/index map layouts; manifest and commit plumbing; workspace refs; the single-writer lock. CLI: `init`, `status`, `commit`, `log`, `branch`, `checkout`, plus low-level `put-node`/`get-node` plumbing commands for testing. Also `fsck` in skeletal form — building the verifier alongside the format pays for itself immediately.

Exit criteria: a scripted end-to-end — init, insert nodes/edges via plumbing, commit, branch, mutate, diff roots — with `git log` and `git push` to a real GitHub private repo working untouched; property-test suite green (history independence, encoding order, merge determinism at the map level); fsck clean.

## Phase 2 — Cypher read path (size L)

`acetone-cypher` front end and read-only execution per spec §5.3: parser adoption (decypher evaluation vs vendored openCypher grammar — decide in week one of the phase), binder against schema, logical plan, iterator-model operators for scan/seek/expand/filter/project/aggregate/sort/limit, `OPTIONAL MATCH`, `UNWIND`, parameters, and openCypher null semantics. Stand up the openCypher TCK harness immediately and drive development from it. CLI: `query` and the `shell` REPL with table/JSON output.

Exit criteria: TCK read-scenario pass rate ≥ a chosen bar (suggest 60% of read features as the honest MVP line, published, with a tracked climb thereafter); a lab asset graph (say, 50k nodes / 200k edges of hosts, software, suppliers, certificates) queryable with realistic registry queries at interactive latency; `AT <ref>` time travel over the Phase 1 commit graph.

## Phase 3 — Cypher write path and the commit discipline (size M)

`CREATE`, `MERGE` (upsert-on-key), `SET`, `REMOVE`, `DELETE`, `DETACH DELETE`; transactional write batching into workspace roots; constraint enforcement (key presence, uniqueness, existence) at write and commit; key-immutability rule and the `rekey` utility. This is the phase where the workbench loop closes: checkout → edit via Cypher → status/diff → commit.

Exit criteria: write-feature TCK scenarios passing for the supported subset; idempotence demonstrated — re-running a `MERGE`-based load of identical data produces an unchanged root and `commit` reports nothing to commit; interactive editing session captured end-to-end in docs.

## Phase 4 — Diff, merge and conflicts (size L)

Graph-level `diff` (classified node/edge change streams and the `_Added`/`_Removed`/`_Modified` virtual graph), merge-base computation over the git commit graph, map-wise three-way merge, post-merge graph validation (dangling edges, constraint re-check), the `conflicts` map, `acetone.conflicts()` and `_Conflict` querying, `resolve`, and merge commits. Property-based testing regime: generate random base graphs and divergent edit scripts; assert merge determinism, symmetry properties where applicable, and that clean merges never produce dangling edges.

Exit criteria: the flagship demo works — two branches import overlapping asset data, merge produces both clean results and representative conflicts, conflicts are inspected and resolved in Cypher, history shows the whole story; blame implemented for nodes.

## Phase 5 — Import/export and secondary indexes (size M)

The `import` plugin interface (CSV and JSON/NDJSON built in; a plugin trait for source-system extractors), deterministic transform convention, provenance trailers, `--branch` import isolation; `export` to CSV/JSON (the seed of the relational projection); declared property indexes with `IndexSeek` planner integration and `reindex`. Round out `gc` and `fsck`.

Exit criteria: a scheduled-import simulation — N successive snapshots of a mutating source imported as commits, with no-op detection on unchanged snapshots and `diff` between runs serving as the change report; index-accelerated query demonstrably faster than scan on the lab graph; fsck verifying index and reverse-edge consistency.

## Phase 6 — Hardening towards 0.1 (size M)

Format freeze review of spec §3.4 encodings and manifest (with `format_version` machinery and a working `migrate`), TCK pass-rate push, error-message quality, docs (user guide, format spec, conformance statement), packaging (single static binary; `acetone-core` published as a library crate), and a dogfood deployment: the real asset-registry use case run in anger on a private GitHub remote.

Exit criteria: 0.1 tagged; a fortnight of dogfooding without data-integrity incidents; the three documents in this pack revised to match reality.

> **Status (2026-07-09).** Landed: the format freeze at `format_version = 1` (ADR-0024, golden-anchored), `.acetone/` tree namespacing (ADR-0023), the `acetone migrate` history-rewrite engine (ADR-0025), the `acetone-core` façade crate + release packaging (ADR-0026), the [user guide](user-guide.md) and [conformance statement](conformance.md) (openCypher pass rate 41.0% / 1596 of 3897, known gaps tracked). Remaining at the boundary: the dogfood *fortnight* (inherently post-boundary), the 0.1 tag (a human call), and continued TCK/error-message improvement filed as beads. See [`reports/phase-6.md`](reports/phase-6.md).

## Phase 7 — Installable and trustworthy (size L)

Reframed from the pre-0.1 review (`reports/pre-0.1-review.md`): 0.1 shipped a rigorous storage engine below the manifest and a younger query/runtime layer above it, and every urgent pre-tag defect lived at that seam. Phase 7 **carries the lower system's invariant discipline upward into the query/runtime layer** and does so dogfood-UX-first — it is the last chance to fix the internal seams before the `acetone-core` library API commits to them at 0.2. Runtime maturation: a **query resource governor** capping execution time, result rows, expansion steps and list size (`acetone-iq6`) — the blocker for any library or embedded use; **lazy, store-backed `IndexSeek`** that touches only matching rows instead of materialising a whole version per statement (`acetone-cbl.11`, `acetone-6g5.3.3`); **stable relationship identity** replacing the positional `e{index}` ids with a real edge identity *before* any API freezes it (`acetone-rid`, root cause shared with the U11 duplicate-edge fix); a **value-domain round-trip contract** — a typed value channel through the query adapter so temporal and `Bytes` values read back as themselves, not debug strings (`acetone-vdc`, of which U2 was the acute symptom); then the **library-level query entry point** and routing the CLI through the façade (`acetone-vf6`, `acetone-ijq`), done *after* the seam fixes so the API doesn't freeze the defects. Merge lifecycle and meaning: **cell-level (per-property) merge** as Decision 4 of the spec promises, so two branches editing different properties of the same node no longer conflict — the common `import sets os_version / human sets owner` case (`acetone-clm`, Greg's call at the demo: implement cell-wise or amend the design); merge abort and mid-merge recovery (`acetone-mws`), post-merge validation semantics (`acetone-jmp`), side-by-side conflict visibility (`acetone-s7d`). Robustness: `fsck` on a repo whose default workspace manifest is damaged (`acetone-zhp`), worktree gc-durability edge cases (`acetone-7tf`, `acetone-cm9`), and an error-message quality pass (`acetone-cbl.3`). Dogfood UX: workspace `diff`/`discard`, property-level structured (JSON) diff for change reports, ancestry refspecs `main~1`/`HEAD^` (`acetone-bvq`), import-vs-curation via import-to-branch + merge (`acetone-6g5.11`, depends on cell-level merge), and export hardening (`acetone-6g5.9`). The dogfood registry runs continuously from 0.1 and ranks this work by what actually hurts.

Exit criteria: `acetone-core` exposes a **governed** library query entry point — bounded execution time, result rows, expansion steps and memory — and the CLI itself runs through it; a property/fuzz regime shows no query can exhaust memory or wall-clock past the caps; relationship identity and the value-domain contract are stable enough to freeze as public API; cell-level merge demonstrated on the divergent-property case; the dogfood registry has run continuously since 0.1 without a data-integrity incident. This is the **0.2 gate** — the point at which the library API stabilises and (optionally) the crates are published bottom-up. (In the event the API froze at 0.2.0 — ADR-0046 — and the publication option was declined: crates.io is held as standing policy, ADR-0047 point 5; see §Distribution.)

## Phase 8 — Alongside code (size M)

Make an acetone graph a **co-tenant of an ordinary git repository** — its own branch, Dolt-style, or embedded beside code. There is no on-disk format obstacle, but four behavioural assumptions must flip *together*, best unified behind one `GraphRefNamespace` concept (`acetone-gns`): `HEAD` currently means "the graph checkout" and must become "the graph's ref, wherever it lives"; `refs/heads/*` is used unqualified and must be namespaced so graph refs and code refs coexist; `migrate` rewrites *all* refs and must rewrite only the graph's; `gc` walks *all* refs for reachability and must scope to the graph's refs, never deleting objects reachable only from code history (`acetone-5w6` co-tenancy constraint, `acetone-ejj` migrate ref-scoping, `acetone-7tf`/`acetone-cm9`/`acetone-060` worktree and HEAD edge cases). The **format-evolution policy is settled before the first format bump**: an ADR (`acetone-fev`) choosing **read-old-write-new** as the default evolution path — retain old-format decoders, no history rewrite (`acetone-5yr`) — and reserving the history-rewriting `migrate` for explicit opt-in, because rewrite is incompatible with the sharing pitch (it force-pushes all history and changes every commit hash).

Exit criteria: an acetone graph lives on its own ref inside a repository that also holds code, with code branches and graph branches coexisting untouched; `migrate` and `gc` provably touch only graph refs (property test: objects reachable only from code history survive `gc`, code refs survive `migrate`); a `format_version` bump is applied to a live graph via read-old-write-new with no history rewrite and no remote force-push.

## Phase 9 — At scale and in conformance (size L)

Push the envelope out and climb the conformance curve. Scale: **streaming / bounded-memory import** for sources larger than memory (`acetone-6g5.7`); `fsck` at scale — dedup chunk sets across versions, walk tags, worktree-aware (`acetone-7fe`, `acetone-8t3`, `acetone-6g5.10`); a `merge_base` LCA bound to remove the worst-case (~cubic) blow-up (`acetone-vgt`). Conformance: climb the published openCypher TCK pass rate (41.0% at 0.1) through pattern comprehension (`acetone-cxh`), label predicate in expression position (`acetone-6gy`, `acetone-q9m`, `acetone-i8z`), the `i64::MIN` literal (`acetone-4lh`) and binder refinements (`acetone-1qj`). Query performance: **composite index seek acceleration** (`acetone-0c7`), `IndexRange` for range predicates, uniqueness/partial indexes (`acetone-ryg`, definition already frozen wide enough per ADR-0027), and the seeds of **costed planning** to replace the current heuristic anchor choice and address var-length path performance.

Exit criteria: a published TCK pass-rate climb against a stated bar, with the conformance statement updated; import of a source larger than memory completes in bounded resident memory; index range and composite seek measurably beat scan on the lab graph at the larger envelope; `fsck` and `merge_base` costs verified sub-quadratic on a multi-version repository.

## Beyond Phase 9 (unscheduled, in rough order of pull)

Views and materialised views over the provider-pluggable scan interface, and watch/reactive queries — all built on the derived-map machinery whose first instances are the secondary indexes; the virtual-element provider API generalised from the conflict/diff machinery; structured change-report export (diff → PR-style artefact) as the natural product of the diff engine; `acetone log`/`acetone blame` over graph history (blame substrate exists, `acetone-596`); schema-evolution tooling (rename label/property) riding on the format-evolution path; RDF projection (oxigraph-backed SPARQL over exports first, live adapter later) and relational/Parquet projection; a native chunk store behind the existing `ChunkStore` trait if scale demands it; `CALL {}` subqueries and GQL-alignment syntax; and an optional read-only server mode for dashboards — a deliberate late arrival, because the workbench identity comes first.

## Distribution and release

0.1 ships as a **single static binary** (`acetone`) — musl-static on Linux, the platform binary on macOS — built by the `Release` GitHub Actions workflow and staged as a **draft** GitHub release; publishing the draft is the human approval and mints the `v<version>` tag (there is no manual tagging). Distribution is a **Homebrew binary formula** in `curvelogic/homebrew-tap` (per-platform `url` + `sha256`). The library crates stayed **internal** through 0.1 — no `crates.io` publication, no frozen external API — which is exactly what kept the Phase-7 seam fixes cheap. The API duly froze at the 0.2 gate (ADR-0046), but publication did not follow: the standing policy (Greg, 2026-07-21; ADR-0047 point 5) is that the crates are **not published to crates.io** until Greg judges the project mature enough, or an external need forces it. If and when that changes, publication is bottom-up in dependency order; until then the frozen surface is documented by `STABILITY.md` rather than docs.rs. See `RELEASING.md`.

## Decision gates and open questions

The four roadmap gates are **closed**: Gate A (git ODB as chunk store) — confirmed at end of Phase 0; Gate B (parser adoption vs vendored grammar) — decided by ADR in Phase 2; Gate C (TCK bar for MVP honesty) — the honest MVP line published in the conformance statement; Gate D (format freeze) — `format_version = 1` frozen and golden-pinned (ADR-0024). The three forward gates named at the Phase 7–9 extension (ADR-0032) are now **closed** too: the **0.2 library-API freeze** — the curated `acetone-core` root surface frozen at 0.2.0, machine-enforced by committed `cargo-public-api` snapshots (ADR-0046, Greg's boundary decision 2026-07-18); the **merge-granularity decision** — cell-wise (per-property) merge, as design-space Decision 4 promised, ratified at the Phase 7 / 0.2 boundary (ADR-0035, `acetone-clm`); and the **format-evolution ADR** — read-old-write-new as the default evolution path with the history-rewriting `migrate` reserved for explicit opt-in, settled before Phase 8's first format bump and ratified at the Phase 8 / 0.3 boundary (ADR-0048, `acetone-fev`). No named forward gates remain; new ones are declared by ADR as they arise.

The Phase 1–2 open questions are **resolved**: SHA-1 remains the default for hosting ubiquity (major forges including GitHub cannot yet host SHA-256 repositories), with `--object-format sha256` as opt-in and hash width validated-not-assumed on read (ADR-0031); time travel is `AT <ref>`, with ancestry refspecs (`main~1`, `HEAD^`) deferred to `acetone-bvq`; edge records are **not** duplicated into `edges_rev` (it is a derived reverse-adjacency map, reproducible by `reindex`); v0.1 has **no parallel edges** — the edge key carries a discriminator slot that is always `Null` from the query surface (ADR-0030). Whether `schema` changes should require their own commits remains open, deferred alongside the schema-evolution tooling beyond Phase 9.

## Suggested repo scaffolding

Cargo workspace `acetone/` with the six crates from spec §8, `tck/` vendoring the openCypher TCK runner, `benches/` holding the Phase 0 suite kept alive as regression benchmarks, and `docs/` holding these three documents as the living design record. CI: test + property-test + TCK-conformance report as a published artefact per commit — a database that version-controls data should be embarrassed by anything less for its own code.
