# 0.1.1 — CLI & Cypher ergonomics sweep

Boundary report for the 0.1.1 release. This is not a roadmap phase; it is a
focused, dogfood-driven ergonomics release cut between Phase 6 (0.1.0) and
Phase 7, on Greg's call that "significant ergonomic improvement may constitute
an 0.1.1 before we move onto Phase 7."

## What it is

0.1.0 shipped as "two systems of different maturity wearing one name": a
rigorous storage engine below the manifest, a younger query/runtime layer
above, and the daily friction all lived at that seam and on the CLI surface.
0.1.1 carries the storage engine's discipline **upward** into the things a
user actually touches — error messages, help, the shell, and machine output —
**without any on-disk format change**. 0.1.0 repositories are read and written
unchanged.

Framing: Phase 7 ("installable and trustworthy") always had two threads —
dogfood-UX polish and runtime hardening toward the 0.2 library-API freeze.
0.1.1 is the UX thread shipped early as a point release; Phase 7 proper remains
the hardening (governor, relationship identity, value-domain contract, the
`acetone-core` facade).

## What shipped

22 beads across 12 PRs (#101–#112), each TDD'd, independently reviewed by a
fresh adversarial subagent with no implementation context, and squash-merged on
green CI + sign-off. Main stayed green throughout.

**Foundation.** A snapshot harness (`insta`) pins every user-facing error and
help string, so wording changes surface as reviewed diffs — the safety net that
made the rest of the sweep regression-safe (7bn.1).

**Error quality.** A single canonical node-key renderer in `acetone-model`
kills the `[String("x")]` Debug leaks and double-`Debug` (7bn.2). Every Cypher
error now reads `line L, column C:` — execution errors gained location, the
redundant `at bytes` byte-offset is gone (7bn.3). `DuplicateKey` advice is
clause-correct and gives the MERGE idiom; missing-colon labels and relationship
types are explained; `declare-*` is suggested (7zw, b96). Unknown label /
property / function / relationship-type errors suggest the nearest declared
name via a new edit-distance module, and the SKIP/LIMIT value leak is fixed
(7bn.4, 7bn.14). Vocabulary is settled to "relationship" at the Cypher boundary
(7bn.6). All identifiers/values reaching the terminal are escaped (hostile
clones cannot inject terminal sequences).

**CLI surface.** Help is concise, grouped by role, and states the three-way
relationship to git; `cypher` aliases `query`; prefixes infer; `--format` is
consistent across `import`/`export`; `get-node` exits non-zero on a miss; and a
no-flag `migrate` preserves the current chunk profile (004, 6bg, ohv, 7bn.7).
`acetone schema [--at <ref>]` displays the declared schema for any version
(7bn.10). A `--json` flag on the read commands makes them scriptable, with the
shape documented as not-yet-frozen (7bn.11). `acetone` works from a
subdirectory, `git -C`-style, preserving the config-isolation boundary (7bn.12,
ADR-0034).

**Shell.** `acetone shell` gains readline editing, history, a branch-aware
prompt, in-shell `:declare-*`/`:commit`/`:status`/`:schema`/`:cancel`,
wide-character-aware table alignment, distinct `NULL`, a row cap, and errors on
stderr — while staying scriptable when piped (86i, 4vy, 7bn.8, 7bn.9,
ADR-0033).

## ADRs taken (for ratification)

- **ADR-0033** — adopt `rustyline` for the interactive shell (a runtime
  dependency, `acetone-cli`-only; TTY-gated with a scriptable fallback; a
  transitive BSL-1.0 licence added to the allow-list).
- **ADR-0034** — discover the repository from a subdirectory, preserving the
  `isolated().with(Trust::Reduced)` config-isolation boundary verbatim.

New dev dependency `insta` (snapshots) and runtime dependency `unicode-width`
(display width, `acetone-cli`-only, already in-tree via rustyline) were adopted
with in-PR justification; both are low-stakes and did not warrant separate
ADRs.

## Review findings summary

The mandatory fresh-subagent review caught real defects the implementer had
missed — the gate earned its keep:

- **A blocker**: the first cut of `migrate`'s default flags would have silently
  rewritten all history to a chunk profile no `init` produces (defaults taken
  from the wrong `ChunkParams::default`). Fixed — a no-flag `migrate` now
  preserves the repo's current params (a hash-preserving repair) — and
  re-reviewed.
- An **unescaped label** reintroduced into a `DuplicateKey` suggestion (the
  exact terminal-injection class the module guards against) — fixed to a
  generic placeholder.
- An **unterminated final statement silently dropped at EOF** in the piped
  shell — fixed to flush on EOF.
- `serde_json` leaving DEL/C1 control bytes raw in `--json` — the implementer
  pre-empted it with a residual-control escape pass.

All other findings were fixed inline or filed as follow-ups.

## Deferred / follow-ups

## Milestone security review

A dedicated security pass over the whole `v0.1.0..HEAD` diff (fresh
strongest-tier subagent). **Gate verdict: READY** — no blocker- or
high-severity findings.

Confirmed clean: every dangerous terminal-control class (C0, C1, DEL,
ANSI/ESC) is neutralised on every new output path (human, `--json`, `schema`,
shell table, prompt, suggestions) — verified by driving hostile bytes end to
end; the `isolated().with(Trust::Reduced)` config-isolation boundary is
preserved through repository discovery, and `init` cannot alias an enclosing
repo; no new `unsafe` (all shipping crates keep `forbid(unsafe_code)`); the new
dependencies (`rustyline`, `unicode-width` runtime; `insta` dev) are confined to
`acetone-cli`/dev and `cargo deny` (advisories + licences) is clean; discovery
and value-encoding paths are panic-free on hostile input. The
`render_node_key`/dangling-edge refactor is a net improvement — it escapes an
endpoint label that 0.1.0 emitted raw.

Three low/informational findings, none gate-blocking. The one security finding
(LOW-1) was pulled into 0.1.1 and fixed at Greg's boundary call; the rest are
filed as follow-ups:

- **Resolved (was LOW-1, `acetone-7bn.19`)** — bidirectional/format control
  characters (`U+061C`, `U+200E–200F`, `U+202A–202E`, `U+2066–2069`) passed raw
  through the sentence-shaped sanitisers (`sanitise_line`/`json_string`/
  `emit_json`), which filtered on `char::is_control()` only. This allowed
  Trojan-source-style *visual* reordering of hostile-clone text (property
  values, commit subjects, branch names) — **not** escape-sequence execution
  (all cursor/colour/clear controls were already escaped). Greg's boundary call
  was to **pull the fix into 0.1.1**: a shared `is_bidi_control` predicate now
  escapes the directional-formatting set on every repository-controlled
  terminal path — the human table/`--format csv`, the branch-aware prompt,
  `status`, the `branch` list, `log`/`:log`, `query --format json`, and the
  `--json` reads — bringing them to the identifier renderer's bar. (The
  mandatory fresh review caught that `status` and the `branch` list still
  printed branch names raw — the exact branch-name vector — and those were
  fixed before merge.) Verified end to end by driving a `U+202E` value through
  the property-value and branch-name paths. Zero-width / other
  Cf characters are deliberately out of scope (they enable invisibility/
  homoglyph confusion, not reordering, and escaping them would mangle
  legitimate emoji ZWJ sequences) — noted as a residual, lesser follow-up.
- INFO (`acetone-7bn.20`) — an optional edit-distance length guard and a
  cosmetic duplicate in the discovery error message.
