The flagship · conflicts as data

A clash is something you query,
not an error you fight.

  • Merge conflicts → the workspace enters merge-in-progress: a persisted conflicts map + a MERGE_HEAD ref.
  • Inspect in Cypher — CALL acetone.conflicts() and the _Conflict virtual subgraph.
  • Resolve — pick a side (resolve --all-ours|--all-theirs) or write a hand-merged value; the write clears the conflict.
  • Finalise — commit writes the two-parent merge. fsck clean throughout.
$ acetone merge other
merge produced 1 conflict(s):
node "N" [1]
$ acetone query "CALL acetone.conflicts()
YIELD node WHERE '_Conflict' IN labels(node)
RETURN node.name"
(:_Conflict:N {name: ours})
$ acetone resolve --all-theirs
resolved 1 conflict(s)
$ acetone commit -m "merge other"
committed 86d5955… (two parents)
04 / 09

Decisions · for your retrospective

Five ADRs. One was your call.

ADR-0016 · post-merge validation

Validation lives in the pure merge core (deterministic, property-covered). Only merge-introduced breaches are reported.

ADR-0017 · procedure seam & the 8c3 split

A ProcedureProvider seam; the virtual-graph surface split out because its invocation was underspecified.

ADR-0018 · diff virtual-graph surface YOUR CALL

CALL … YIELD node chosen over stateful CALL/MATCH and AT-range — most openCypher-standard, non-stateful.

ADR-0019 · blame follows first-parent

Git --first-parent semantics — deterministic, O(log n) per commit.

ADR-0020 · merge-in-progress state

A MERGE_HEAD ref + a conflicts index that stores which keys clash, not their values — re-derived from ours/theirs. Only cell conflicts persist.

Live demo · one step per turn

Driving the real CLI.

Each step is backed by a passing test, so the demo can't drift from the code. We run them one at a time and stop to discuss.

STEP 1 · 2

Branch & disjoint edits: seed a graph, branch other, edit both sides apart, merge → a clean two-parent commit; diff and log tell the story.

STEP 3

Overlapping edits → conflict: both sides edit the same node; merge reports it; status shows merge-in-progress.

STEP 4

Inspect in Cypher: CALL acetone.conflicts() and the _Conflict subgraph — the clash as queryable nodes.

STEP 5

Resolve & complete: resolve --all-theirs or write a merged value, then commit — a two-parent merge, fsck clean.

STEP 6

Blame: CALL acetone.blame('N', k) attributes the node's history across the whole merge.

THEN

Your call: rule on the ADRs and the graph-violation scope, then close the Phase 4 gate bead acetone-14c.7.
