3/11 · the format

Node identity has exactly one byte form, everywhere

A node key — (primary label, key tuple) — encodes as a single memcomparable list element, and those exact bytes appear standalone as the nodes-map key and embedded inside edge and index keys (ADR-0008). Identity comparisons are byte comparisons in every map; label scans are prefix scans.

("Host", "web1")  →  0c 06 48 6f 73 74 … fb 06 77 65 62 31 … fb 00
nodes map key         └── the same bytes ──┘
edges_fwd key   [src node key][type][dst node key][disc]
  • Key properties live only in the key — a record that disagrees with its key is unrepresentable (Invariant 3 enforced structurally).
  • The manifest is version-first: [format_version, body], so any future reader identifies a newer repository before parsing a byte of body.
  • All of it golden-pinned: changing any byte is a declared format_version bump, not an accident.

4/11 · the repository

An acetone repo is a bare git repo you can read with git

a commit, in git's eyes

commit fe0083c…
├── README.md      (human summary)
├── manifest       (the version record)
└── chunks/        (anchors every chunk)

git log, git show, hosting UIs, git fsck --strict: all just work. Trailers carry provenance.

the moving parts (ADR-0010)

  • Workspace = manifest blob at refs/acetone/workspaces/<name>, advanced by compare-and-swap — a lost race is a typed error, never a lost write.
  • Single-writer lock held for a whole transaction; readers are pinned to immutable manifests (MVCC for free).
  • Real git HEAD is the checked-out ref — clones see the right branch.

Proven, not claimed: a committed 500-node version survives git gc --prune=now --aggressive and a cold reopen — commits anchor the complete chunk set of every map, so whole versions travel with clone/push/fetch.

6/11 · storage economy

Pack-on-write, part 1 — the problem, and the bet

Every 1%-churn import commit rewrites the O(log n) spine of each touched key. Chunk version N+1 is almost identical to version N — a prolly node with one entry changed — but git stores them as two independent blobs.

~39MiB retained per
1%-churn commit (1M keys)
~17×the actual
changed payload
~17min for plain git gc
at ~900k loose objects

git gc reclaims some, but it is slow at that scale and picks delta bases by a size/similarity guess that knows nothing about the tree.

The bet (validated in Phase 0, bead 63m.10, at 7.1× better retention): we already know each new chunk's predecessor — so hand git the exact delta base instead of making it guess.

Two foundations make that safe. Predecessor recording is a pure observer on apply_batch: it receives (new chunk → predecessor) after each chunk is stored, and the tree built is byte-identical whether or not you are watching — history independence (Invariant 1) must not depend on observation. And the representation-only invariant: consolidation changes how bytes are stored, never which bytes and never any hash — a chunk's git OID is the same loose, packed whole, or stored as a delta.

7/11 · storage economy

Pack-on-write, part 2 — consolidation

acetone gc takes the accumulated loose chunks plus the predecessor hints and rewrites them into one pack with chosen delta bases.

the four load-bearing mechanics

  • Bases-first, iterative — each base precedes its deltas; iterative not recursive (a recursive walk stack-overflowed at 100k chains, caught in review), duplicate OIDs and cycles handled.
  • Chain caps — a whole object ~every 32 versions bounds reconstruction (>6× vs 7.1×; reads stay fast).

.

  • Prune gated on the pack — delete only OIDs the written pack's index contains; a hard error if it misses the reachable set, so a planning bug fails loud, never deletes the last copy.
  • fsync before prune — pack + idx + dir durable before any unlink; a dedicated acetone-gc.lock keeps commits unblocked.

Every emitted delta is round-trip-verified against the exact chunk bytes and falls back to a whole object on any doubt (base past git's u32 offset, no size win, verify fail). Payoff: 7.1× (Phase 0); the regression benchmark that keeps it honest is bead acetone-1qw. Run a foreign git gc only after committing — let acetone gc do the packing.

8/11 · integrity

fsck — born paranoid

Built alongside the format, not after it: a verifier for every version a repository can reach, on the assumption that repositories arrive over the network from hostile origins.

  • Verifies every version reachable from workspaces, branches and tags: manifest decode, chunk reachability, and prolly structure checked at each chunk's claimed position in the tree — a chunk valid in isolation but referenced from the wrong place is caught, not trusted.
  • MISSING and CORRUPT distinguished; every finding names its offending chunk.
  • An advisory tier for later-phase semantics (edge symmetry, annotated tags) — incompleteness is declared in the report, never silent.
  • Cross-version work memoised on (root, height) so a hostile repo of many tiny commits can't force O(history × tree) re-verification.

The hardest thing to get right in a verifier is that it never says clean when the data isn't. Four review rounds; two proven false-cleans — fsck reporting clean while get/scan reject the same repo — found and killed before merge, one of them introduced by an earlier review fix (next slide).

10/11 · decisions & risks

Five ADRs on record; open risks are beaded and bounded

decisions taken (made, not deferred)

  • 0008 map layouts + manifest encoding · 0010 workspace refs, writer lock, HEAD semantics · 0011 pack-on-write consolidation (includes a deliberate spec §3.1 amendment) · 0012 fsck taxonomy · 0009 subagent model tiers (yours).

open risks (none blocker-class)

  • P2 write_ref create-CAS accepts a value-equal no-op (acetone-0ej) · fsck can't run if the default workspace manifest itself is the damaged object (acetone-zhp) · fsck tag-peeling and cross-version dedup (acetone-8t3/7fe).
  • Format-level gap, documented: uncommitted workspaces don't survive a foreign git gc until acetone-gc protects them (ADR-0010).
  • P3s: checkout two-ref atomicity, library no-change-commit guard, consolidation scaling, anchor-completeness check, and friends — all beaded.
