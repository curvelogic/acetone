Acetone · Sprint demo · Phase 8 boundary

Alongside code.

Until now an acetone repository was the graph — it owned HEAD, refs/heads/*, every object. Phase 8 makes a graph a co-tenant of an ordinary git repository: its own ref, living beside your code, in one object store. Code branches and graph branches coexist — and neither gc nor a format bump ever touches the other's history.

Where we are

No format obstacle — four behavioural assumptions.

Storing a graph beside code needs no new on-disk format. What stood in the way were four assumptions baked into the ref/store layer, each true only because the repo was the graph. They had to flip together, or a half-converted repo corrupts one tenant to serve the other.

HEAD
the graph checkout
→ the graph's ref, wherever it lives
refs/heads/*
unqualified
→ namespaced, so graph + code coexist
migrate
rewrites all refs
→ only the graph's
gc
walks all refs
→ never prunes code-only history

The wrong shape is two divergent code paths — standalone and co-tenant drifting apart bug by bug. Phase 8 flips all four behind a single value, so there is one code path and only the layout differs.

02 / 13

The one concept · ADR-0049

One value maps logical refs to physical paths.

GraphRefNamespace is the seam every ref-path decision now passes through. A Repository builds one at init/open; the code never branches on "mode", only the namespace value differs. Two layouts, one path.

STANDALONE · the default, forever

The repo is the graph

Branches refs/heads/*, tags refs/tags/*, head pointer git HEAD. Clone-shows-graph-on-main ergonomics, unchanged. The gns refactor is byte-identical — every store/graph/CLI test and every pinned chunk golden passed untouched.

CO-TENANT · new in 0.3

The graph is a guest

Branches refs/heads/acetone/<graph>/*, tags refs/tags/acetone/<graph>/*, and the graph's current-branch pointer at a local-only symref refs/acetone/<graph>/HEAD. Your git HEAD and code branches are never touched.

proxy-safe  The acetone/ prefix lives under refs/heads — a subnamespace proxies allow, where a top-level refs/acetone/* would 403 (operational-constraints note).

Exit criterion 1 · acetone-mgf (#147)

MET

A graph on its own ref, code untouched beside it.

Init opts a graph into co-tenant mode, drops a marker, and open-time detection selects the co-tenant namespace. The result: two tenants' refs in one repository, visibly distinct, mutually undisturbed.

git for-each-ref — one repo, two tenants
$ git for-each-ref --format='%(refname)'
refs/heads/main ← your code, on the shared HEAD
refs/heads/feature/parser ← your code branch
refs/heads/acetone/assets/main ← the graph, namespaced
refs/acetone/assets/HEAD ← graph's branch pointer (local-only)
$ git symbolic-ref HEAD
refs/heads/main ✓ your checkout, never moved

Co-tenant init is wired at the library layer; the acetone init --co-tenant CLI flag is a tracked follow-up (acetone-xg6).

Exit criterion 2 · acetone-iva (#148) · ADR-0051

MET — with a reading to ratify

gc and migrate stay in the graph's lane.

The co-tenancy promise that matters: the graph never rewrites or moves your code history. Proven by property tests — a code-only object survives gc, code refs survive migrate — with a discriminator that only passes if the reachable set genuinely includes code.

PROVEN

Code survives both

An object reachable only from code history is still retrievable after acetone gc; every code ref is intact after migrate. No code ref moves; no commit hash changes.

THE FRESH-REVIEW CATCH

A vacuous test, hardened

The first gc proof would have passed even if scoping were wrong. The review forced a loose-file discriminator: a code blob is drawn into gc's pack — consolidated yet still retrievable — which only holds if gc sees code.

⚑ Reading (A), shipped: acetone gc repacks the whole repo's reachable set, like git gc — byte-preserving, fsck-clean, nothing moved. This is the recorded interpretation of "touch only graph refs". Ruling on slide 11.

Exit criterion 3 · acetone-5yr (#149) · ADR-0052

MET — machinery + coexistence proven

Evolve the format without rewriting shared history.

The manifest envelope is the stable [format_version, body]: read the version first. So decode now dispatches to a retained per-version decoder instead of rejecting anything non-current. Old commits read through their era's decoder; new writes emit the current format; a repository holds both, side by side, with nothing rewritten.

read-old-write-new — one store, two versions
# a content-addressed store holds a v1 commit AND a v2 commit
decode(v1_bytes) → Manifest ✓ via the retained v1 decoder
decode(v2_bytes) → Manifest ✓ via the v2 decoder
blob_hash(v1) unchanged by the v2 write ✓ no rewrite
re-encode(v1) → same bytes, same address ✓ no force-push
production build (v1-only) decode(v2) → UnsupportedVersion(2)

FORMAT_VERSION stays 1 — zero format change, zero golden churn. The v2 above is a test-only synthetic; a real v2 is deferred to the first change that warrants one. Ruling on slide 11.

The forward gate · ADR-0048

Two evolution strategies; the default is the safe one.

Crossing a format boundary can be done two ways. They pull in opposite directions, so Phase 8 settled which is the default before the first bump could land.

READ-OLD-WRITE-NEW · default

Retain decoders, never rewrite

The binary keeps a reader per shipped format and dispatches on version. Old commits stay put; new writes use the current format. No hash changes, no force-push — the only strategy compatible with a repo shared with code and collaborators.

REWRITE-MIGRATE · opt-in

Bring history up to current

Re-encodes every reachable commit — which changes every hash and needs a force-push. Fine for a standalone repo that is the graph and wants a single-format history or to reclaim old-object space. A deliberate act, never automatic.

acetone offers both; the choice is the operator's, defaulting to the non-destructive one. Nothing on the normal read/write path ever rewrites history to cross a format boundary.

Two rulings · your call at the boundary

Where the recorded reading needs your sign-off.

ADR-0051 · co-tenant gc

How wide does gc reach?

  • (A) shipped — repack the whole repo's reachable set, code included, like git gc. Byte-preserving, fsck-clean, nothing moved. Simple; whole-repo maintenance.
  • (B) alternative — scope gc's pack to the graph's refs while still treating code refs as un-prunable roots. Stronger isolation; a more intricate reachability split. No format impact.

Recommend (A) — and confirm it as the reading of exit criterion 2's gc half.

ADR-0052 · exit criterion 3

Is a synthetic v2 enough for 0.3?

  • Shipped — the dispatch machinery + a test-only synthetic v2 proving coexistence. FORMAT_VERSION stays 1; zero golden churn. A real v2 lands through this seam at the first genuine format change.
  • Alternative — require a real format_version 2 to ship in 0.3 (re-pins every manifest/commit golden; changes the on-disk format purely to demonstrate).

Recommend shipped — the ADR-0025 "engine now, real use deferred" precedent you accepted at 0.2.
