1/7 · sprint demo · 4 July 2026

acetone

A version-controlled labelled property graph database: Dolt-style prolly trees stored in a real git repository, queried with openCypher, run as a workbench, written in Rust.

The whole architecture rests on one bet — Decision 1, Option A: that the git object database can be the chunk store, so clone, push, hosting, review and CI come for free. Phase 0 existed solely to test that bet with running code before anything was built on it.

This deck covers Phase 0 (complete, gate ratified) and the opening of Phase 1 (in flight). Then: a live demo, then the docs are yours.

4/7 · the number to watch

History growth under churn is the honest cost

17×payload retained per 1%-churn
import commit (~39 MiB each)
17 mingit gc at ~918k
accumulated loose objects

Random 1% churn rewrites ~36% of leaf chunks per commit, and git's pack heuristics cannot delta content-addressed blobs — aggressive repack recovered only ~1% more than plain gc.

The proposed fix is an unvalidated hypothesis. Pack-on-write with explicitly chosen delta bases (we know each chunk's predecessor at write time) could recover most of it — but no existing git or gix tooling writes custom-delta packs, so it means hand-rolling git's delta and pack formats.

That experiment (acetone-63m.10) is running right now; success bar ≥5× improvement with git-readable packs. Fallbacks if it fails: import key locality, coarser batching, history compaction.

6/7 · decisions

Made, ratified, and open

Ratified by you

  • Gate A: git ODB as chunk store — go (ADR-0002)
  • Licence: MIT OR Apache-2.0 (ADR-0003)
  • MPL-2.0: uluru-scoped exception — since made moot: feature-trimmed gix dropped it entirely
  • Review class: executable config = governing docs

Taken by agents (ADR'd)

  • Build acetone-prolly from scratch; prollytree crate disqualified with evidence
  • gitoxide-only write path, gix ≥ 0.85
  • Encoding format decisions (ADR-0004): chunked 8+1 string framing, NaN rejected in keys, −0.0 → +0.0, tags 74100–74102

Open — need you

  • NaN in indexes: float-NaN properties can't produce an index key; skip-from-index aligns with Cypher comparison semantics but ORDER-BY-via-index would never surface those rows (bead 6g5.3, needed by Phase 5)
  • Upstream gix bug report: outward-facing, awaiting your go-ahead (bead 63m.12)

Consequences flagged forward

  • Key order deliberately diverges from openCypher ORDER BY (no Int/Float interleave; Null first) — Phase 2 needs its own comparator; recorded on the executor bead
